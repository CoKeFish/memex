# Plan de prueba E2E — Dashboard memex

> **Objetivo:** recorrer todas las vistas y opciones del dashboard con el MCP de **chrome-devtools**
> para **encontrar errores** — datos incorrectos, requests fallidos, errores de consola, layout roto,
> spinners colgados, estados vacíos mal manejados.
>
> **Autorización:** permiso total para interactuar con todo, incluidas acciones que mutan datos o
> llaman APIs externas, con **tope de USD 2** de gasto de LLM de la app (DeepSeek). Frenar las
> acciones marcadas 💸 `cost-llm` al acercarse al tope.

## Cómo levantar la app
1. **Backend (Docker):** `doppler run -p shared -c dev -- docker compose up -d` → Postgres `5454`,
   MinIO `9000`/`9001`, API `8787`. (Doppler inyecta las llaves de LLM/ingesta al contenedor.)
2. **Frontend (Vite):** `cd frontend && npm run dev` → `http://localhost:5173/` (proxy `/api` → `:8787`).
3. Abrir `http://localhost:5173/`. En dev la auth está **apagada** (`GET /auth/me` devuelve un usuario
   por defecto) → entra directo a `/`. Si redirige a `/login`, registrarse una vez (email + contraseña ≥8).

## Metodología por vista (chrome-devtools)
1. `navigate_page` a la ruta → `wait_for` contenido → `take_snapshot` (árbol con uids).
2. Recorrer cada control: leer su estado, interactuar (`click` / `fill` / `hover`), observar el resultado.
3. Tras cada interacción: `list_console_messages` (errores JS) + `list_network_requests` (4xx/5xx).
4. `take_screenshot` ante cualquier anomalía; anotar el bug con pasos de repro + la request/consola.
5. Marcar `[x]` lo que pasa; dejar nota al lado de lo que falla.

## Leyenda de sensibilidad
- 🟢 `safe` — solo lectura / navegación / UI local. Probar libremente.
- 🟠 `mutates` — crea/edita/activa datos de la app (reversible). Probar con cuidado.
- 🔴 `destructive` — borra o cambia datos de forma irreversible. Usar un dato de prueba.
- 💸 `cost-llm` — dispara un LLM pagado de la app. **Cuenta contra el tope USD 2.**
- 🌐 `external-api` — llama a un proveedor externo / trae datos reales (IMAP/Telegram/Apify/Google).
